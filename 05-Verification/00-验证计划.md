# 验证计划

> [!abstract] 概述
> 验证计划（Verification Plan）定义了验证目标、策略、资源和时间表，是确保芯片功能完整验证的关键文档。

---

## 验证计划内容

### 1. 项目概述

```markdown
项目名称: [项目代号]
芯片类型: [SoC/ASIC/FPGA]
设计规模: [门数/晶体管数]
工艺节点: [工艺信息]
目标应用: [应用场景]
```

### 2. 功能规格分析

#### 2.1 设计特性清单

```markdown
| 特性 ID | 功能描述 | 优先级 | 验证方法 | 状态 |
|---------|----------|--------|----------|------|
| F001    | AXI4 主接口 | P0 | Directed/Constrained Random | Done |
| F002    | DDR 控制器 | P0 | Coverage Driven | Done |
| F003    | 中断处理 | P1 | Coverage Driven | WIP |
```

#### 2.2 接口列表

```markdown
| 接口名称 | 协议 | 方向 | 带宽需求 | 验证优先级 |
|----------|------|------|----------|------------|
| axi_mast | AXI4 | Master | 4GB/s | P0 |
| apb_slv  | APB | Slave | 10MB/s | P1 |
```

### 3. 验证策略

#### 3.1 验证层次

```
系统级 (System Level)
    └── 芯片级 (Chip Level)
        └── 模块级 (Module Level)
            └── 子模块级 (Sub-module Level)
```

#### 3.2 验证方法选择

| 方法 | 适用场景 | 工具/语言 |
|------|----------|-----------|
| 黑盒验证 | 快速功能检查 | SV/UVM |
| 白盒验证 | 内部逻辑验证 | SV + SVA |
| 灰盒验证 | 接口协议验证 | UVM + Assertions |
| 随机约束验证 | 复杂场景 | CVC/SystemVerilog |

#### 3.3 验证平台架构

```verilog
// UVM 测试平台架构
class chip_env extends uvm_env;
    `uvm_component_utils(chip_env)

    axi_env   m_axi_env;
    apb_env   m_apb_env;
    ddr_env   m_ddr_env;
    scoreboard m_sb;
    coverage   m_cov;

    function void build_phase(uvm_phase phase);
        m_axi_env = axi_env::type_id::create("m_axi_env", this);
        m_apb_env = apb_env::type_id::create("m_apb_env", this);
        m_ddr_env = ddr_env::type_id::create("m_ddr_env", this);
        m_sb     = scoreboard::type_id::create("m_sb", this);
        m_cov    = coverage::type_id::create("m_cov", this);
    endfunction
endclass
```

### 4. 验证点列表

#### 4.1 基本功能点

```markdown
- [ ] 上电复位功能
- [ ] 时钟管理
- [ ] 寄存器读写
- [ ] 中断生成和清除
- [ ] 数据通路正确性
```

#### 4.2 协议符合性

```markdown
- [ ] AXI4 读交易正确性
- [ ] AXI4 写交易正确性
- [ ] AXI4 突发传输
- [ ] AXI4 乱序/Outstanding
- [ ] APB 状态机转换
```

#### 4.3 边界条件

```markdown
- [ ] 边界值测试 (0, -1, MAX)
- [ ] 超时处理
- [ ] 错误注入恢复
- [ ] 并发访问冲突
```

#### 4.4 压力测试

```markdown
- [ ] 最大带宽测试
- [ ] 长时间稳定性
- [ ] 多接口并发
```

### 5. 覆盖率目标

| 覆盖率类型 | 目标 | 实际 | 差距 |
|------------|------|------|------|
| 功能覆盖率 | 100% | 98% | 2% |
| 代码覆盖率 | 95% | 90% | 5% |
| 断言覆盖率 | 90% | 85% | 5% |

#### 6.1 人力安排

```markdown
| 角色 | 人数 | 主要职责 |
|------|------|----------|
| DV Lead | 1 | 验证计划、RTL 审查 |
| Verification Engineer | 3 | 验证环境开发、用例编写 |
| Verification Assistant | 2 | 用例执行、结果分析 |
```

#### 6.2 工具清单

```markdown
| 工具 | 版本 | 用途 |
|------|------|------|
| Questa/Verdi | 2023.x | 仿真、调试 |
| VIP | - | 标准协议验证 |
| 覆盖率工具 | - | 覆盖率收集分析 |
```

#### 6.3 时间安排

```markdown
| 阶段 | 开始日期 | 结束日期 | 里程碑 |
|------|----------|----------|--------|
| 验证计划编写 | Week 1 | Week 2 | VP Approved |
| 环境开发 | Week 2 | Week 6 | Env Ready |
| 用例开发 | Week 4 | Week 12 | TB Complete |
| 回归测试 | Week 10 | Week 16 | Final Signoff |
```

### 7. 风险评估

| 风险项 | 影响 | 概率 | 缓解措施 |
|--------|------|------|----------|
| 设计变更频繁 | 高 | 中 | 早期介入、持续沟通 |
| 覆盖率收敛困难 | 中 | 高 | 提前识别corner case |
| 资源不足 | 中 | 低 | 预留buffer |

### 8. 验收标准

```markdown
## Sign-off Criteria

1. **功能覆盖率** >= 100%
2. **代码覆盖率**:
   - Branch Coverage >= 95%
   - Condition Coverage >= 90%
   - FSM Coverage >= 100%
3. **所有 P0/P1 缺陷已关闭**
4. **回归测试通过率** = 100%
5. **随机种子已固定，结果可复现**
```

## 验证计划评审检查项

> [!warning] 评审要点
> 在验证计划评审会上，逐项检查以下内容是否完整、合理。

### 完整性检查

- [ ] 所有设计特性是否已列出且分配优先级
- [ ] 所有接口是否已识别并确定验证方法
- [ ] 覆盖率目标是否明确且可量化
- [ ] 验收标准是否清晰、可执行
- [ ] 资源估算是否合理（人力、工具、时间）

### 策略合理性检查

- [ ] 验证层次划分是否合理（模块级/芯片级/系统级）
- [ ] 随机约束 vs 定向测试的比例是否适当
- [ ] 是否考虑了边界条件和异常场景
- [ ] 是否有压力测试和长时间稳定性测试计划
- [ ] 参考模型的精度是否满足需求

### 覆盖率检查

- [ ] 功能覆盖率是否覆盖所有功能点
- [ ] 代码覆盖率目标是否达到行业标准 (Branch >= 95%)
- [ ] 是否有断言覆盖率计划
- [ ] 覆盖率收敛策略是否明确

### 风险管理检查

- [ ] 是否识别了关键技术风险
- [ ] 缓解措施是否具体可行
- [ ] 是否有设计变更的应对方案
- [ ] 是否有资源不足的应急预案

### 流程合规检查

- [ ] 是否符合公司验证流程规范
- [ ] 是否与设计团队对齐了接口定义
- [ ] 是否有定期进度跟踪机制
- [ ] 是否有缺陷管理流程

## 实际案例参考

> [!example] SPI 控制器验证计划示例
> 以下是一个简化的 SPI 控制器验证计划，供参考。

### 项目信息

```
项目名称: SPI_Controller_Verification
设计规模: ~5K gates
接口: APB (配置) + SPI (数据)
目标: 验证 SPI 主/从模式功能正确性
```

### 功能特性清单

| 特性 ID | 功能描述 | 优先级 | 验证方法 |
|---------|----------|--------|----------|
| F001 | APB 寄存器读写 | P0 | Directed |
| F002 | SPI Master 发送 | P0 | Constrained Random |
| F003 | SPI Master 接收 | P0 | Constrained Random |
| F004 | SPI Slave 发送/接收 | P0 | Constrained Random |
| F005 | 时钟极性/相位配置 (CPOL/CPHA) | P1 | Directed |
| F006 | 多 CS 片选管理 | P1 | Constrained Random |
| F007 | FIFO 溢出/下溢处理 | P1 | Error Injection |
| F008 | 中断生成与清除 | P2 | Coverage Driven |

### 覆盖率计划

```
功能覆盖率:
  - 所有 CPOL/CPHA 组合 (4 种)
  - 所有传输长度 (1-32 字节)
  - Master/Slave 模式切换
  - FIFO 状态组合

代码覆盖率:
  - Branch Coverage >= 95%
  - FSM Coverage = 100%
  - Toggle Coverage >= 90%
```

### 验证环境架构

```
                    spi_sequence
                         ↓
                    spi_sequencer
                         ↓
    spi_driver ←── get_next_item ──→ spi_transaction
         ↓
       DUT (SPI Controller)
         ↓
    spi_monitor ──→ scoreboard
         ↓
    spi_coverage
```
