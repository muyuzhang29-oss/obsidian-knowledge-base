# UVM 验证环境模板总览

> 以 SPI 协议为示例，展示标准 UVM 验证环境的完整架构

---

## 模板使用指南

> [!tip] 快速上手
> 按照以下步骤使用模板搭建你的 UVM 验证环境。

### 使用步骤

```
步骤 1: 确定协议和接口
├── 分析 DUT 接口协议 (AXI/APB/SPI/I2C/...)
├── 确定需要的 Agent 数量和方向 (Active/Passive)
└── 列出所有需要验证的功能点

步骤 2: 复制并修改模板
├── 复制整个模板目录
├── 将 spi_ 前缀替换为你的协议名 (如 axi_, i2c_)
├── 修改 interface 中的信号定义
└── 修改 transaction 中的字段定义

步骤 3: 实现各组件
├── 01-interface: 添加实际信号和时钟
├── 02-transaction: 定义输入/输出字段
├── 03-sequence: 编写激励生成逻辑
├── 04-driver: 实现信号驱动逻辑
├── 05-monitor: 实现信号采集逻辑
├── 06-reference_model: 实现期望值计算
├── 07-scoreboard: 实现比对逻辑
└── 08~11: 组装环境并编写测试

步骤 4: 验证与调试
├── 编译检查语法错误
├── 运行 smoke test 验证基本功能
├── 添加覆盖率收集
└── 逐步扩展测试场景
```

### 文件替换示例

```systemverilog
// 原始模板 (SPI)
class spi_transaction extends uvm_sequence_item;
    rand bit [7:0] cmd;
    rand bit [31:0] addr;
    // ...

// 替换为 AXI
class axi_transaction extends uvm_sequence_item;
    rand bit [31:0] addr;
    rand bit [31:0] data;
    rand bit [7:0]  len;
    // ...
```

### 扩展 Agent 类型

```systemverilog
// Passive Agent: 只有 Monitor，用于协议检查
class axi_passive_agent extends uvm_agent;
    axi_monitor m_monitor;

    function void build_phase(uvm_phase phase);
        super.build_phase(phase);
        m_monitor = axi_monitor::type_id::create("m_monitor", this);
        // 不创建 driver 和 sequencer
    endfunction
endclass

// Active Agent: 有 Driver + Sequencer + Monitor
class axi_active_agent extends uvm_agent;
    axi_driver    m_driver;
    axi_sequencer m_sequencer;
    axi_monitor   m_monitor;

    function void build_phase(uvm_phase phase);
        super.build_phase(phase);
        m_monitor   = axi_monitor::type_id::create("m_monitor", this);
        m_driver    = axi_driver::type_id::create("m_driver", this);
        m_sequencer = axi_sequencer::type_id::create("m_sequencer", this);
    endfunction
endclass
```

### 覆盖率集成示例

```systemverilog
// 在 Monitor 中添加覆盖率
class spi_monitor extends uvm_monitor;
    covergroup spi_cov;
        cp_cmd: coverpoint tr.cmd {
            bins write = {8'h01};
            bins read  = {8'h02};
        }
        cp_len: coverpoint tr.data_len {
            bins short = {[1:4]};
            bins medium = {[5:16]};
            bins long  = {[17:32]};
        }
        cmd_x_len: cross cp_cmd, cp_len;
    endgroup

    function new(string name, uvm_component parent);
        super.new(name, parent);
        spi_cov = new();
    endfunction

    virtual task run_phase(uvm_phase phase);
        forever begin
            // ... 采集 transaction
            spi_cov.sample();
        end
    endtask
endclass
```

## 常见问题解答

> [!question] FAQ

### Q1: Driver 中 `get_next_item` 和 `item_done` 的调用时机？

```
正确模式:
    forever begin
        seq_item_port.get_next_item(req);  // 阻塞等待
        drive_transaction(req);             // 驱动信号
        seq_item_port.item_done();          // 通知 sequencer
    end

常见错误:
    - 忘记调用 item_done() → sequencer 永久阻塞
    - 在 get_next_item 之前操作信号 → 驱动无效数据
```

### Q2: config_db 的 set/get 路径如何匹配？

```
set 路径 = 相对路径 + 通配符
get 路径 = 相对路径

示例:
    // 在 test 中 set
    uvm_config_db#(virtual spi_if)::set(this, "m_env.m_agent*", "vif", vif);

    // 在 driver 中 get (相对于 driver 自身)
    uvm_config_db#(virtual spi_if)::get(this, "", "vif", vif);

注意:
    - set 的第一个参数是上下文 (通常是 this 或 null)
    - set 的第二个参数是相对于第一个参数的路径
    - get 的路径是相对于 get 调用者的路径
```

### Q3: 如何实现 Sequence 的随机约束？

```systemverilog
class spi_sequence extends uvm_sequence#(spi_transaction);
    // 定义约束变量
    rand int unsigned num_transactions;

    constraint c_num {
        num_transactions inside {[10:100]};
    }

    virtual task body();
        repeat(num_transactions) begin
            `uvm_do_with(req, {
                req.cmd inside {8'h01, 8'h02};
                req.data_len inside {[1:32]};
            })
        end
    endtask
endclass
```

### Q4: Scoreboard 如何处理乱序响应？

```systemverilog
// 使用队列存储期望值，按 ID 匹配
class spi_scoreboard extends uvm_scoreboard;
    spi_transaction exp_queue[$];

    virtual function void compare(spi_transaction actual);
        int idx = -1;
        // 查找匹配的期望值
        foreach(exp_queue[i]) begin
            if (exp_queue[i].id == actual.id) begin
                idx = i;
                break;
            end
        end

        if (idx >= 0) begin
            // 比对并移除
            compare_fields(exp_queue[idx], actual);
            exp_queue.delete(idx);
        end else begin
            `uvm_error("SCB", $sformatf("No matching expected for id=%0d", actual.id))
        end
    endfunction
endclass
```

### Q5: 如何调试 UVM 环境？

```
常用调试方法:
1. 增加 UVM 详细级别: +UVM_VERBOSITY=UVM_HIGH
2. 使用波形查看信号时序
3. 在关键位置添加 `uvm_info 打印
4. 使用 objection 机制追踪 phase 结束时机
5. 检查 factory 注册是否正确

调试命令:
    # 编译时
    xrun -uvm -uvmhome $UVM_HOME ...

    # 仿真时
    ./simv +UVM_VERBOSITY=UVM_HIGH +UVM_TIMEOUT=100000
```

## 架构图

```
                    sequence
                       ↓
                   sequencer
                       ↓
    driver ←──── get_next_item ────→ transaction (输入字段)
       ↓                ↓
      DUT          driver.ap ──→ ref_model (读输入字段计算期望) ──→ exp_trans ──→ scoreboard
       ↓
    monitor ──→ rx_trans (输出字段: 实际值) ──→ scoreboard
```

---

## 数据流

```
1. driver 读 transaction 输入字段 → 驱动到 vif → DUT
2. driver 通过 ap 广播 transaction → ref_model 收到输入激励
3. DUT 产生输出
4. monitor 从 vif 采集输出 → 填入 rx_trans 输出字段 → scb
5. ref_model 读输入字段 → 计算期望值 → 填入 exp_trans 输出字段 → scb
6. scb 比对 rx_trans vs exp_trans 的输出字段
```

## Transaction 设计

| 分类 | 字段 | 谁填 | 用途 |
|------|------|------|------|
| 输入 | cmd, addr, data[], data_len, rd_en, rd_len | driver | 驱动 DUT |
| 输出 | status_o, data_o[], error_o | monitor 填实际值 | DUT 实际响应 |
| 输出 | status_o, data_o[], error_o | ref_model 填期望值 | 期望响应 |

**同一个 transaction 类，不同组件用不同部分。**
