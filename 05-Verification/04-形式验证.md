# 形式验证（Formal Verification）

> [!abstract] 概述
> 形式验证是一种**数学证明**方法，通过穷举所有可能的输入组合和状态空间来证明设计的正确性，无需依赖测试向量。与仿真验证互补，是现代芯片验证流程中不可或缺的环节。

---

## 1. 形式验证 vs 仿真验证

| 维度 | 形式验证 | 仿真验证 |
|------|----------|----------|
| 验证原理 | 数学证明，穷举所有状态 | 激励驱动，采样检查 |
| 完备性 | 可证明属性对所有输入成立 | 只能证明已测试的场景 |
| 状态空间 | 受限于设计复杂度（状态爆炸） | 理论上无上限，但受限于仿真时间 |
| 调试能力 | 提供反例路径（Counter-example） | 提供波形和日志 |
| 适用范围 | 局部属性、控制逻辑、协议检查 | 全芯片级、数据通路、复杂场景 |
| 投入产出 | 前期投入大，后期维护成本低 | 前期投入小，回归测试成本高 |
| 典型场景 | 接口协议、FIFO、仲裁器、FSM | SoC 集成、端到端数据流、软件协同 |

### 何时选择形式验证

- 需要**穷举证明**的关键属性（如互斥、无死锁）
- 仿真难以覆盖的 corner case（如并发访问冲突）
- 控制逻辑密集、状态空间有限的模块
- 安全关键设计的合规性证明

### 何时依赖仿真验证

- 大规模数据通路和复杂算法
- 需要软件协同的系统级场景
- 状态空间过大导致形式验证不可收敛

## 2. 形式验证类型

### 2.1 模型检查（Model Checking）

自动穷举有限状态机的所有可达状态，验证属性是否成立。

**原理：**
- 将设计建模为有限状态机（FSM）
- 将属性表达为时序逻辑公式（CTL/LTL）
- 算法自动遍历状态空间，找到反例或证明属性成立

**优点：** 全自动化，无需人工引导；发现 Bug 时提供最小反例路径
**缺点：** 状态爆炸问题，只适用于有限状态系统

```verilog
// 模型检查典型属性示例
// 1. 安全性：某状态不会被到达
assert property (@(posedge clk) disable iff (!rst_n)
    !(state == ERROR_STATE));

// 2. 活性：请求最终会得到响应
assert property (@(posedge clk) disable iff (!rst_n)
    req |-> s_eventually ack);

// 3. 互斥：两个信号不会同时有效
assert property (@(posedge clk)
    !(rd_en && wr_en));
```

### 2.2 等价性检查（Equivalence Checking）

比较两个设计描述在功能上是否等价，常用于 RTL vs RTL、RTL vs Netlist 的比对。

**类型：**

| 类型 | 用途 | 说明 |
|------|------|------|
| 组合等价性（DEC） | 组合逻辑比对 | 逐信号比较组合逻辑锥 |
| 时序等价性（SEC） | 时序逻辑比对 | 需要状态映射关系 |

**典型应用场景：**
- 综合前后 RTL vs Netlist 签核
- ECO（工程变更）后确认功能不变
- 低功耗插入 isolation/power switch 后的功能一致性
- DFT 扫描链插入后确认逻辑等价

```
RTL (Golden)  ──→  综合  ──→  Netlist (Revised)
                    ↓
            等价性检查工具
                    ↓
              PASS / FAIL + Counter-example
```

### 2.3 定理证明（Theorem Proving）

将设计和属性表达为数学定理，由交互式或自动定理证明器推导证明。

**特点：**
- 理论上可处理无限状态空间
- 需要大量人工交互和专家知识
- 适用于高度抽象的算法正确性证明

**常用工具：** ACL2、Coq、Isabelle/HOL

**应用场景：** 浮点运算单元的 IEEE 754 合规性、加密算法的数学正确性

## 3. SVA 在形式验证中的应用

SVA（SystemVerilog Assertions）是形式验证中最常用的属性描述语言。与仿真中的断言不同，形式验证中的 SVA 被**穷举证明**而非随机采样检查。

### 3.1 属性分类

| 属性类型 | 含义 | SVA 关键字 | 说明 |
|----------|------|------------|------|
| 安全性（Safety） | 坏事永远不发生 | `assert property` | 如：不会溢出、不会死锁 |
| 活性（Liveness） | 好事最终会发生 | `assert property` + `s_eventually` | 如：请求最终被响应 |
| 覆盖性（Coverage） | 某场景可达 | `cover property` | 如：所有状态都可被访问 |

### 3.2 形式验证中的典型 SVA 属性

```verilog
// ---- 安全性属性 ----

// FIFO 不会溢出
property p_fifo_no_overflow;
    @(posedge clk) disable iff (!rst_n)
    (count == DEPTH) |-> !wr_en || rd_en;
endproperty

// 独热码状态机不进入非法状态
property p_onehot_valid;
    @(posedge clk) disable iff (!rst_n)
    $onehot(state);
endproperty

// ---- 活性属性 ----

// 请求最终会得到应答（形式验证中的 liveness property）
property p_req_eventually_acked;
    @(posedge clk) disable iff (!rst_n)
    req |-> s_eventually ack;
endproperty

// 仲裁器不会饿死任何一个请求者
property p_no_starvation;
    @(posedge clk) disable iff (!rst_n)
    req[0] |-> s_eventually gnt[0];
endproperty

// ---- 覆盖属性 ----

// 确认满状态可达
cover property (@(posedge clk) full);

// 确认所有状态都被访问过
cover property (@(posedge clk) state == S_IDLE);
cover property (@(posedge clk) state == S_ACTIVE);
cover property (@(posedge clk) state == S_DONE);
```

### 3.3 形式验证对 SVA 的特殊要求

- **可综合性：** 属性必须可被形式引擎解析，避免使用 `$display`、`$random` 等仿真系统函数
- **有界性：** 有界模型检查（BMC）只能证明有限深度内的属性，需设置足够的深度
- **约束充分性：** 输入约束必须准确，过松导致误报，过紧导致漏报
- **状态空间控制：** 复杂属性可能触发状态爆炸，需拆分或抽象

## 4. 属性规范语言（PSL vs SVA）

### 4.1 PSL（Property Specification Language）

PSL 是 Accellera 标准的语言，语法独立于 HDL，支持 VHDL、Verilog、SystemVerilog。

```
-- PSL 语法示例（VHDL 风格）
-- 请求后 1~3 周期内必须有应答
assert always (req -> next[1..3](ack));

-- 互斥属性
assert always !(rd_en and wr_en);

-- 覆盖属性
cover {req; ack};
```

### 4.2 对比

| 特性 | SVA | PSL |
|------|-----|-----|
| 标准来源 | IEEE 1800 (SystemVerilog) | IEEE 1850 |
| 语言绑定 | SystemVerilog 原生 | 独立于 HDL |
| 时序操作符 | `##N`, `[*N]`, `\|->`, `\|=>` | `next[N]`, `[*N]`, `->` |
| 工具支持 | 所有主流 EDA 工具 | 主流工具均支持 |
| 生态丰富度 | UVM 集成紧密，社区活跃 | 较少使用，但在形式验证工具中广泛支持 |
| 学习曲线 | 需要 SystemVerilog 基础 | 语法更独立，可单独学习 |

**选择建议：** 如果团队已使用 SystemVerilog/UVM，优先选择 SVA；如果需要跨语言复用属性，考虑 PSL。

│                                                                 │
└─────────────────────────────────────────────────────────────────┘

### 关键步骤说明

#### 5.1 约束编写（Constraints / Assumptions）

约束定义了形式验证引擎的搜索空间，必须准确反映设计的运行环境。

```verilog
// 时钟约束：单时钟域
assume property (@(posedge clk) 1'b1);  // 时钟始终存在

// 复位约束：复位期间不进行操作
assume property (@(posedge clk) !rst_n |-> !req && !wr_en);

// 输入协议约束：valid 必须伴随 ready
assume property (@(posedge clk) disable iff (!rst_n)
    valid |-> ready);

// 协议约束：AXI burst length 不超过 16
assume property (@(posedge clk) disable iff (!rst_n)
    arvalid |-> (arlen < 16));
```

#### 5.2 黑盒建模

对于不验证的子模块或外部 IP，用简化的抽象模型替代。

```verilog
// Memory 黑盒模型：只建模读写行为，不建模内部实现
module memory_model (
    input  logic        clk,
    input  logic        wr_en,
    input  logic [7:0]  addr,
    input  logic [31:0] wdata,
    output logic [31:0] rdata
);
    logic [31:0] mem [0:255];

    always_ff @(posedge clk) begin
        if (wr_en)
            mem[addr] <= wdata;
    end

    assign rdata = mem[addr];
endmodule
```

## 7. 形式验证案例

### 7.1 FIFO 正确性验证

验证一个同步 FIFO 的关键属性，确保在所有场景下行为正确。

```verilog
module fifo_formal (
    input  logic        clk, rst_n,
    input  logic        wr_en, rd_en,
    input  logic [7:0]  wdata,
    output logic [7:0]  rdata,
    output logic        full, empty,
    output logic [3:0]  count
);

    parameter DEPTH = 16;
    parameter WIDTH = 8;

    // ---- 输入约束 ----
    // 写满时不写入（来自上层协议）
    assume property (@(posedge clk) disable iff (!rst_n)
        full |-> !wr_en);

    // 空时不读取（来自下层协议）
    assume property (@(posedge clk) disable iff (!rst_n)
        empty |-> !rd_en);

    // ---- 安全性属性 ----

    // P1: 计数器不会溢出
    property p_count_no_overflow;
        @(posedge clk) disable iff (!rst_n)
        count <= DEPTH;
    endproperty
    assert property (p_count_no_overflow);

    // P2: 满标志正确性
    property p_full_correct;
        @(posedge clk) disable iff (!rst_n)
        full <-> (count == DEPTH);
    endproperty
    assert property (p_full_correct);

    // P3: 空标志正确性
    property p_empty_correct;
        @(posedge clk) disable iff (!rst_n)
        empty <-> (count == 0);
    endproperty
    assert property (p_empty_correct);

    // P4: 满时不写入
    property p_no_wr_when_full;
        @(posedge clk) disable iff (!rst_n)
        full |-> !wr_en;
    endproperty
    assert property (p_no_wr_when_full);

    // P5: 空时不读取
    property p_no_rd_when_empty;
        @(posedge clk) disable iff (!rst_n)
        empty |-> !rd_en;
    endproperty
    assert property (p_no_rd_when_empty);

    // ---- 活性属性 ----

    // P6: 有空间时写入最终成功
    property p_wr_eventually;
        @(posedge clk) disable iff (!rst_n)
        (wr_en && !full) |=> (count == $past(count) + 1);
    endproperty
    assert property (p_wr_eventually);

    // P7: 有数据时读取最终成功
    property p_rd_eventually;
        @(posedge clk) disable iff (!rst_n)
        (rd_en && !empty) |=> (count == $past(count) - 1);
    endproperty
    assert property (p_rd_eventually);

    // ---- 覆盖属性 ----
    cover property (@(posedge clk) full);
    cover property (@(posedge clk) empty);
    cover property (@(posedge clk) wr_en && rd_en);  // 同时读写

endmodule
```

**验证结果分析：**

| 属性 | 状态 | 说明 |
|------|------|------|
| P1: count 无溢出 | PROVEN | 通过 |
| P2: full 标志正确 | PROVEN | 通过 |
| P3: empty 标志正确 | FAIL | 反例：复位释放后第一个周期 count 未清零 |
| P4: 满时不写入 | PROVEN | 输入约束保证 |
| P5: 空时不读取 | PROVEN | 输入约束保证 |
| P6: 写入最终成功 | PROVEN | 通过 |
| P7: 读取最终成功 | PROVEN | 通过 |

### 7.2 握手协议验证

验证一个 Valid-Ready 握手协议的正确性。

```verilog
module handshake_formal (
    input  logic        clk, rst_n,
    input  logic        valid, ready,
    input  logic [31:0] data
);

    // ---- 输入约束 ----

    // valid 一旦置起，在 ready 握手之前不能撤销
    assume property (@(posedge clk) disable iff (!rst_n)
        (valid && !ready) |=> valid);

    // valid 撤销后至少保持 1 周期无效
    assume property (@(posedge clk) disable iff (!rst_n)
        $fell(valid) |=> !valid [*1:3]);

    // ---- 安全性属性 ----

    // P1: 数据在 valid 期间必须稳定
    property p_data_stable_when_valid;
        @(posedge clk) disable iff (!rst_n)
        (valid && !ready) |=> $stable(data);
    endproperty
    assert property (p_data_stable_when_valid);

    // P2: 数据只在握手成功时被采样
    property p_data_sampled_on_handshake;
        @(posedge clk) disable iff (!rst_n)
        (valid && ready) |-> $changed(data) or $stable(data);
    endproperty
    assert property (p_data_sampled_on_handshake);

    // ---- 活性属性 ----

    // P3: valid 最终会被 ready 响应（不饿死）
    property p_no_starvation;
        @(posedge clk) disable iff (!rst_n)
        valid |-> s_eventually ready;
    endproperty
    assert property (p_no_starvation);

    // ---- 覆盖属性 ----

    cover property (@(posedge clk) valid && ready);  // 正常握手
    cover property (@(posedge clk) valid && !ready);  // 等待状态

endmodule
```

### 7.3 状态机验证

验证一个三段式 FSM 的状态转换正确性和安全性。

```verilog
module fsm_formal (
    input  logic       clk, rst_n,
    input  logic       start, done, error,
    output logic [1:0] state
);

    localparam S_IDLE   = 2'b00;
    localparam S_ACTIVE = 2'b01;
    localparam S_DONE   = 2'b10;
    localparam S_ERROR  = 2'b11;

    // ---- 安全性属性 ----

    // P1: 状态独热码检查（适用于独热编码 FSM）
    // property p_onehot_state;
    //     @(posedge clk) $onehot(state);
    // endproperty
    // assert property (p_onehot_state);

    // P2: 不进入非法状态（二进制编码下 2'b11 可能非法）
    property p_no_illegal_state;
        @(posedge clk) disable iff (!rst_n)
        !(state == S_ERROR);
    endproperty
    assert property (p_no_illegal_state);

    // P3: 状态转换合法性
    property p_idle_to_active;
        @(posedge clk) disable iff (!rst_n)
        (state == S_IDLE && start) |=> (state == S_ACTIVE);
    endproperty
    assert property (p_idle_to_active);

    property p_active_to_done;
        @(posedge clk) disable iff (!rst_n)
        (state == S_ACTIVE && done) |=> (state == S_DONE);
    endproperty
    assert property (p_active_to_done);

    property p_done_to_idle;
        @(posedge clk) disable iff (!rst_n)
        (state == S_DONE) |=> (state == S_IDLE);
    endproperty
    assert property (p_done_to_idle);

    // P4: 复位后回到 IDLE
    property p_reset_to_idle;
        @(posedge clk) !rst_n |=> (state == S_IDLE);
    endproperty
    assert property (p_reset_to_idle);

    // P5: error 信号在 ACTIVE 阶段才有效
    property p_error_only_in_active;
        @(posedge clk) disable iff (!rst_n)
        error |-> (state == S_ACTIVE);
    endproperty
    assert property (p_error_only_in_active);

    // ---- 覆盖属性 ----
    cover property (@(posedge clk) state == S_IDLE);
    cover property (@(posedge clk) state == S_ACTIVE);
    cover property (@(posedge clk) state == S_DONE);
    cover property (@(posedge clk) state == S_ACTIVE ##[1:$] state == S_DONE);

endmodule
```

## 8. 形式验证的局限性

### 8.1 状态空间爆炸

形式验证的核心限制是**状态空间爆炸问题**。随着设计规模增长，状态空间呈指数增长。

| 设计规模 | 状态空间大小 | 形式验证可行性 |
|----------|-------------|----------------|
| 小型模块（<100 寄存器） | 2^100 | 直接验证 |
| 中型模块（100~1000 寄存器） | 2^1000 | 需要抽象/分解 |
| 大型模块（>1000 寄存器） | 2^1000+ | 通常不可行 |

**缓解策略：**
- **设计分解：** 将大模块拆分为小模块分别验证
- **数据通路抽象：** 将数据通路抽象为黑盒，只验证控制逻辑
- **参数化验证：** 先用小参数验证，再通过归纳证明推广

### 8.2 属性编写的挑战

- **过松的属性：** 可能通过验证但实际存在 Bug（漏报）
- **过紧的属性：** 可能在合法场景下也报告失败（误报）
- **不完整的属性集：** 未覆盖所有关键行为

### 8.3 不适用于的场景

- **模拟电路和混合信号：** 连续值域无法离散化穷举
- **复杂数据算法：** 如浮点除法、FFT 等，状态空间过大
- **软件行为：** 处理器执行的软件路径不可穷举
- **跨时钟域信号：** 需要专门的 CDC 分析工具

### 8.4 形式验证不能替代仿真

```
形式验证  ──→  局部属性的数学证明（深度）
仿真验证  ──→  全局场景的随机覆盖（广度）

最佳实践 = 形式验证（关键属性）+ 仿真验证（系统级场景）
```
