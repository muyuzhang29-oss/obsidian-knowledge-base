# 覆盖率收集实践

> 模块级验证中覆盖率收集的策略、陷阱与最佳实践

---

## 一、覆盖率类型

| 类型 | 说明 | 谁负责 |
|------|------|--------|
| **功能覆盖率** | 验证计划中的功能点是否被覆盖 | 验证工程师手写 covergroup |
| **代码覆盖率** | 工具自动收集（行/分支/FSM/条件/ toggle） | 仿真工具自动收集 |
| **断言覆盖率** | SVA bind 的 cover 属性 | 仿真工具自动收集 |

---

## 二、Toggle 覆盖率的分层策略

### 2.1 问题：配置寄存器的 toggle 覆盖率

模块级验证中，部分信号是通过寄存器配置写入的（如 `reg_spis_ctrl_len_ind`、`reg_src_addr` 等）。这些信号在模块级仿真中：

- **写入后保持不变**，toggle 覆盖率永远达不到 100%
- **不是模块的功能逻辑**，toggle 翻转由寄存器扫描模块负责

```
┌─────────────────────────────────┐
│         SoC 顶层                │
│                                  │
│  ┌──────────┐   ┌────────────┐ │
│  │ reg_scan │──→│ spi_parse  │ │
│  │ (配置写入)│   │ (功能逻辑) │ │
│  └──────────┘   └────────────┘ │
│       ↑                          │
│  toggle在这里收集               │
│  配置信号的翻转由reg_scan保证   │
└─────────────────────────────────┘
```

### 2.2 解决方案：排除配置信号的 toggle 覆盖率

在模块级验证中，排除配置寄存器的 toggle 覆盖率：

```tcl
# VCS 排除 toggle
-toggle -exclude {tb_top.dut.reg_spis_ctrl_len_ind}
-toggle -exclude {tb_top.dut.reg_src_addr}
-toggle -exclude {tb_top.dut.reg_src_port}
-toggle -exclude {tb_top.dut.reg_spis_rw_intv}
-toggle -exclude {tb_top.dut.reg_spis_qos}

# 或者用文件批量排除
-toggle -exclude -file toggle_exclude.list
```

```tcl
# toggle_exclude.list
tb_top.dut.reg_spis_ctrl_len_ind
tb_top.dut.reg_src_addr
tb_top.dut.reg_src_port
tb_top.dut.reg_spis_rw_intv
tb_top.dut.reg_spis_rw_intv_ind
tb_top.dut.reg_spis_qos
tb_top.dut.reg_dst_addr*
tb_top.dut.reg_dst_port*
```

### 2.3 各模块的覆盖率职责

| 模块 | toggle 覆盖率范围 | 说明 |
|------|-------------------|------|
| reg_scan | 配置寄存器信号 | 保证所有配置位都能翻转 |
| spi_parse | 功能逻辑信号 | 排除配置寄存器 |
| 顶层 | 全部信号 | 集成级验证 |

## 三、功能覆盖率设计

### 3.1 覆盖点规划

```verilog
class spi_parse_coverage extends uvm_component;
    `uvm_component_utils(spi_parse_coverage)

    spi_parse_config cfg;
    uvm_analysis_imp #(spi_parse_trans, spi_parse_coverage) imp;

    covergroup cg_frame with function sample(spi_parse_trans tr);
        // 命令类型
        cmd_type: coverpoint tr.cmd[4:3] {
            bins write   = {2'b01};
            bins rd_cmd  = {2'b10};
            bins rd_data = {2'b11};
        }

        // 数据长度
        data_len: coverpoint tr.data_len_l[7:0] {
            bins zero     = {0};
            bins short    = {[1:8]};
            bins medium   = {[9:64]};
            bins long     = {[65:255]};
        }

        // 错误注入
        err_inject: coverpoint {tr.inject_cmd_loc_err, tr.inject_cmd_start_err,
                                tr.inject_cmd_len_err, tr.inject_spis_psr_tout_err} {
            bins no_err    = {4'b0000};
            bins loc_err   = {4'b1000};
            bins strt_err  = {4'b0100};
            bins len_err   = {4'b0010};
            bins tout_err  = {4'b0001};
        }

        // 交叉覆盖
        cmd_x_len: cross cmd_type, data_len;
    endgroup

    function new(string name, uvm_component parent);
        super.new(name, parent);
        cg_frame = new();
    endfunction

    function void write(spi_parse_trans tr);
        cg_frame.sample(tr);
    endfunction
endclass
```

### 3.2 覆盖率收集的常见陷阱

| 陷阱 | 说明 | 解决 |
|------|------|------|
| 采样时机不对 | 在 monitor 采样前就 sample | 在 scb 收到 rx_trans 后采样 |
| 覆盖点太细 | 每个信号都做 coverpoint | 只覆盖验证计划要求的功能点 |
| 忽略交叉覆盖 | 只看单维度 | 关键场景做 cross |
| 覆盖率 100% 就停 | 不等于验证充分 | 结合功能覆盖率 + 代码覆盖率判断 |

## 四、代码覆盖率分析

### 4.1 各类代码覆盖率

| 类型 | 含义 | 关注点 |
|------|------|--------|
| **行覆盖** | 每行代码是否执行 | 死代码检测 |
| **分支覆盖** | if/else 是否都走过 | 条件组合遗漏 |
| **条件覆盖** | 每个条件的 T/F | 复合条件拆分 |
| **FSM 覆盖** | 状态机的状态和跳转 | 未覆盖的状态/跳转 |
| **Toggle 覆盖** | 信号 0→1 / 1→0 | 未翻转的信号 |

### 4.2 覆盖率提升策略

```
代码覆盖率低 → 分析未覆盖的代码
    ↓
├── 死代码 → 确认是否可以删除
├── 防御性代码 → 可以排除（如 default 分支）
├── 缺少激励 → 补充 sequence
└── 条件组合遗漏 → 补充功能覆盖率
```

## 五、覆盖率驱动的验证收敛

### 5.1 收敛流程

```
1. 制定验证计划 → 定义功能覆盖点
2. 搭建环境 → 集成覆盖率模型
3. 运行回归 → 收集功能 + 代码覆盖率
4. 分析漏洞 → 补充 sequence / 调整约束
5. 覆盖率达标 → 验证收敛
```

### 5.2 达标标准（参考）

| 类型 | 模块级 | SoC 级 |
|------|--------|--------|
| 行覆盖率 | ≥ 95% | ≥ 90% |
| 分支覆盖率 | ≥ 90% | ≥ 85% |
| FSM 覆盖率 | 100% 状态覆盖 | ≥ 95% |
| 功能覆盖率 | 100% 覆盖点 | 100% 关键场景 |
| Toggle 覆盖率 | 排除配置信号后 ≥ 90% | ≥ 85% |

## 六、覆盖率排除文件模板

```tcl
# coverage_exclude.tcl

# 防御性 default 分支
-instance tb_top.dut -line *default*

# 配置寄存器 toggle（由 reg_scan 模块负责）
-toggle -exclude {tb_top.dut.reg_*}

# 复位逻辑（低频翻转）
-toggle -exclude {tb_top.dut.rst_*}

# 未使用的保留位
-toggle -exclude {tb_top.dut.reserved_*}
```
