# 寄存器（Register）与锁存器（Latch）

> 核心区别：寄存器是边沿触发，锁存器是电平敏感

---

## 基本概念

### 寄存器（Register / Flip-Flop）

边沿触发的时序逻辑元件，只在时钟的有效边沿（上升沿或下降沿）采样输入并更新输出。

```verilog
// 综合出寄存器
always_ff @(posedge clk or negedge rst_n) begin
    if (!rst_n)
        q <= 1'b0;
    else
        q <= d;
end
```

**特点**：
- 时钟边沿时刻一次性采样输入
- 在两个时钟沿之间，输出保持不变
- 天然具有同步特性，易于时序分析

### 锁存器（Latch）

电平敏感的存储元件，当使能信号为有效电平时透明（输出跟随输入），使能信号无效时锁存（输出保持）。

```verilog
// 综合出锁存器
always_latch begin
    if (en)
        q <= d;    // en=1 时透明，q 跟随 d
    // en=0 时隐式保持，q 不变
end
```

**特点**：
- 使能期间输出实时跟随输入
- 使能结束后锁存最后的值
- 没有时钟边沿概念，是电平驱动

---

## 关键区别对比

| 特性 | 寄存器（Register） | 锁存器（Latch） |
|------|-------------------|----------------|
| **触发方式** | 边沿触发（posedge/negedge） | 电平敏感（en=1 透明） |
| **输出变化时机** | 仅在时钟有效边沿 | 使能有效期间随时变化 |
| **时序分析** | STA 友好，建立/保持时间明确 | STA 复杂，透明窗口难以约束 |
| **面积** | 较大（需要时钟树） | 较小（无时钟树） |
| **功耗** | 时钟翻转时消耗动态功耗 | 使能期间持续消耗功耗 |
| **设计可控性** | 高，同步设计的基石 | 低，容易产生毛刺传播 |
| **典型编码** | `always_ff @(posedge clk)` | `always_latch` 或 `if` 不完整 |

## 综合后的电路影响

### 寄存器综合结果

```
         ┌─────────────┐
  D ────>│             │────> Q
         │  D-Flip-Flop│
  CLK ──>│             │
         └─────────────┘
```

- 综合工具生成标准单元库中的 **DFF**（D 触发器）
- 自动插入时钟树（clock tree）
- 建立时间（setup）和保持时间（hold）由工具自动分析
- 可以插入 scan chain 做 DFT

### 锁存器综合结果

```
         ┌─────────────┐
  D ────>│             │────> Q
         │    Latch    │
  EN ──>│             │
         └─────────────┘
```

- 综合工具生成标准单元库中的 **LAT**（锁存器）
- **不插入时钟树**，EN 信号走数据路径
- 透明窗口内的时序路径难以用 STA 约束
- DFT 插入更复杂

---

## 为什么锁存器不受欢迎

### 1. 时序分析困难

寄存器的时序路径有明确的起点和终点（一个时钟沿到下一个时钟沿），STA 工具可以精确计算 setup/hold。

锁存器的透明窗口意味着：信号可以在使能有效的整个期间传播到输出，STA 工具必须考虑"时间借用"（time borrowing），分析复杂度大幅上升。

### 2. 毛刺传播

使能期间，输入的任何毛刺都会直接传播到输出：

```
EN=1 期间：
  D: ──┐  ┌──┐  ──
       └──┘  └──
  Q: ──┐  ┌──┐  ──     ← 毛刺直接穿透
       └──┘  └──
```

寄存器只在边沿采样，天然过滤了边沿之间的毛刺。

### 3. 综合警告

综合工具（Genus、DC）遇到非预期的 latch 推断会报 warning：

```
Warning: Incomplete if/else/case statement infers latch.
```

这是最常见的 latch 产生原因——**组合逻辑中条件分支不完整**。

## 编码行为与综合结果

### 一、产生寄存器的编码模式

> 规则：只要 `always` 块的敏感列表包含时钟边沿（`posedge`/`negedge`），综合结果就是寄存器。

#### 1. 标准 always_ff

```verilog
// ✅ 综合出寄存器，带同步复位
always_ff @(posedge clk) begin
    if (rst)
        q <= '0;
    else
        q <= d;
end
```

#### 2. 带异步复位的寄存器

```verilog
// ✅ 综合出寄存器，带异步复位
always_ff @(posedge clk or negedge rst_n) begin
    if (!rst_n)
        q <= '0;
    else
        q <= d;
end
```

#### 3. 使能控制的寄存器

```verilog
// ✅ 仍然是寄存器，只是多了一个 en 门控
always_ff @(posedge clk) begin
    if (en)
        q <= d;
    // en=0 时 q 保持不变 → 这是寄存器行为，不是 latch
end
```

> [!important] 关键区别
> 在 `always_ff @(posedge clk)` 中，即使 `if` 没有 `else`，综合结果也是寄存器（保持上一个值），**不会**推断出 latch。Latch 只在组合逻辑块（`always_comb`）中因不完整分支产生。

#### 4. 移位寄存器

```verilog
// ✅ N 级移位寄存器
always_ff @(posedge clk) begin
    sr <= {sr[N-2:0], din};
end
```

#### 5. 计数器

```verilog
// ✅ 计数器本质是寄存器 + 组合逻辑反馈
always_ff @(posedge clk) begin
    if (rst)
        cnt <= '0;
    else
        cnt <= cnt + 1;
end
```

#### 6. 多个赋值取最后一个

```verilog
// ✅ 仍然是寄存器，后面的赋值覆盖前面的
always_ff @(posedge clk) begin
    q <= a;      // 这条被覆盖
    q <= b;      // 最终综合出 q <= b
end
```

---

### 二、产生锁存器的编码模式

> 规则：在组合逻辑块（`always_comb`/`always @*`）中，某个信号在某些条件下没有被赋值，综合工具必须保持它的值 → 推断 latch。

#### 1. if 缺少 else

```verilog
// ❌ out 在 sel=0 时需要保持 → 推断 latch
always_comb begin
    if (sel)
        out = a;
    // 没有 else
end

// ✅ 补全 else
always_comb begin
    if (sel)
        out = a;
    else
        out = '0;
end
```

#### 2. case 缺少 default

```verilog
// ❌ sel=2'b11 时 out 没有赋值 → 推断 latch
always_comb begin
    case (sel)
        2'b00: out = a;
        2'b01: out = b;
        2'b10: out = c;
    endcase
end

// ✅ 补全 default
always_comb begin
    case (sel)
        2'b00: out = a;
        2'b01: out = b;
        2'b10: out = c;
        default: out = '0;
    endcase
end
```

#### 3. 嵌套条件不完整

```verilog
// ❌ 外层 if 有 else，但内层 if 缺少 else → b_out 推断 latch
always_comb begin
    a_out = '0;
    if (sel_a) begin
        a_out = data;
        if (sel_b)           // 内层 if
            b_out = data;    // sel_b=0 时 b_out 没有赋值
    end
end

// ✅ 内层也补全
always_comb begin
    a_out = '0;
    b_out = '0;
    if (sel_a) begin
        a_out = data;
        if (sel_b)
            b_out = data;
        else
            b_out = '0;
    end
end
```

#### 4. case 分支内赋值不完整

```verilog
// ❌ 2'b01 分支只赋值了 out_a，out_b 没有赋值 → out_b 推断 latch
always_comb begin
    out_a = '0;
    out_b = '0;
    case (sel)
        2'b00: begin
            out_a = a;
            out_b = b;
        end
        2'b01: begin
            out_a = c;
            // out_b 在这个分支没有赋值
        end
        default: begin
            out_a = '0;
            out_b = '0;
        end
    endcase
end
```

#### 5. 敏感列表不完整（旧式写法）

```verilog
// ❌ 敏感列表缺少 sel → 综合工具认为这是组合逻辑但信号不全 → latch
always @(a, b) begin
    if (sel)
        out = a;
    else
        out = b;
end

// ✅ 用 always_comb 自动推断完整敏感列表
always_comb begin
    if (sel)
        out = a;
    else
        out = b;
end
```

#### 6. for 循环内条件赋值不完整

```verilog
// ❌ match=0 时 out 没有赋值 → 推断 latch
always_comb begin
    out = '0;
    for (int i = 0; i < 8; i++) begin
        if (data[i] == target)
            out = i[2:0];    // 多次匹配时取最后一个值
    end
end
// 注意：这个例子实际上不会产生 latch，因为 out 有默认值
// 但如果去掉 out = '0 就会
```

#### 7. always_latch 显式声明

```verilog
// 显式声明锁存器，这是有意为之
always_latch begin
    if (en)
        q <= d;
end
```

### 三、易混淆的边界情况

#### 1. always_ff 中 if 缺少 else → 寄存器，不是 latch

```verilog
// ✅ 这是寄存器！不是 latch！
always_ff @(posedge clk) begin
    if (en)
        q <= d;
    // en=0 时 q 保持上一个值 → 寄存器行为
end
```

时钟边沿触发的块中，所有信号在每个边沿都有确定的行为（保持或更新），不需要 else 来避免 latch。

#### 2. 带默认值的组合逻辑 → 不产生 latch

```verilog
// ✅ 有默认值，所有路径都有赋值 → 纯组合逻辑，无 latch
always_comb begin
    out = '0;               // 默认值覆盖所有未赋值的情况
    if (cond_a) out = a;
    if (cond_b) out = b;
    if (cond_c) out = c;
end
```

#### 3. unique/priority case → 不产生 latch

```verilog
// ✅ unique case 要求完备覆盖，工具会检查并报错
always_comb begin
    unique case (sel)
        2'b00: out = a;
        2'b01: out = b;
        2'b10: out = c;
        2'b11: out = d;   // 必须覆盖所有值
    endcase
end
```

#### 4. 完整的 if-else if 链 → 不产生 latch

```verilog
// ✅ 最后有 else，所有路径都有赋值
always_comb begin
    if (sel == 2'b00)
        out = a;
    else if (sel == 2'b01)
        out = b;
    else if (sel == 2'b10)
        out = c;
    else
        out = d;
end
```

#### 5. 三目运算符 → 不产生 latch

```verilog
// ✅ 三目运算符天生完备，不会产生 latch
assign out = sel ? a : b;
```

### 四、快速判断表

| 编码行为 | 综合结果 | 说明 |
|---------|---------|------|
| `always_ff @(posedge clk)` | 寄存器 | 无论 if/case 是否完备 |
| `always_comb` + 完整 if-else | 纯组合逻辑 | 无寄存器无 latch |
| `always_comb` + 完整 case+default | 纯组合逻辑 | 无寄存器无 latch |
| `always_comb` + if 缺少 else | **锁存器** | 某些条件下信号需要保持 |
| `always_comb` + case 缺少 default | **锁存器** | 未覆盖的 case 值需要保持 |
| `always_comb` + 有默认值 + 不完整 if | 纯组合逻辑 | 默认值兜底，无 latch |
| `always @(posedge clk or negedge rst_n)` | 寄存器 | 旧式写法，同 always_ff |
| `always @*` + 不完整分支 | **锁存器** | 同 always_comb |
| `always_latch` | **锁存器** | 显式声明 |
| `assign out = sel ? a : b` | 纯组合逻辑 | 三目运算符天然完备 |

### 五、消除 latch 的万能写法

```verilog
// 方法 1：给信号赋默认值（推荐）
always_comb begin
    out = '0;               // 默认值
    if (cond) out = a;      // 条件覆盖
end

// 方法 2：完整 if-else
always_comb begin
    if (cond)
        out = a;
    else
        out = '0;
end

// 方法 3：完整 case + default
always_comb begin
    case (sel)
        2'b00: out = a;
        2'b01: out = b;
        default: out = '0;
    endcase
end

// 方法 4：先赋默认值再用 case 覆盖（最安全）
always_comb begin
    out = '0;
    case (sel)
        2'b00: out = a;
        2'b01: out = b;
        // 不需要 default，前面的 out = '0 已经兜底
    endcase
end
```

## 什么时候可以接受锁存器

| 场景 | 说明 |
|------|------|
| **时钟门控（Clock Gating）** | 用 latch 做 ICG（Integrated Clock Gating），是标准做法 |
| **总线保持** | 某些总线协议需要在特定期间保持上一个值 |
| **面积优化** | 对时序不敏感的低功耗设计中，latch 面积更小 |
| **Time Borrowing** | 高性能设计中利用 latch 的透明窗口做时间借用 |

> [!warning] 关键原则
> 除非你是有意使用锁存器（intentional latch），否则综合出 latch 基本都是 bug。用 `always_comb` + 完整的 if/case 分支可以避免意外推断。

## 验证中的影响

### 对 Testbench 的影响

- **仿真行为不同**：锁存器在使能期间输出会实时变化，可能导致 monitor 采到中间值
- **时序违例风险**：STA 无法完全覆盖的路径可能在 silicon 上失败
- **X 态传播**：锁存器的初始值不确定（无复位时），可能导致仿真中 X 态传播

### 检查方法

```verilog
// 综合后检查是否有非预期 latch
// Genus: report_cells -hierarchical -filter "is_latch == true"
// DC:    report_cell [get_cells -hierarchical -filter "is_latch == true"]
```
