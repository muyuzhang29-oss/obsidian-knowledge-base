# 时序分析基础

> [!abstract] 概述
> 时序分析是数字电路设计与验证的核心环节。验证工程师不仅需要验证功能正确性，还必须确保设计在目标频率下满足所有时序约束。本笔记从验证工程师视角系统讲解时序分析的原理、方法与实践。

---

## 1. 时序分析的重要性

### 为什么验证工程师要懂时序

很多验证工程师认为"时序是后端的事"，这是一个常见误区：

| 场景 | 不懂时序的后果 |
|------|---------------|
| 仿真模型选择 | 使用 `#0` 延迟模型，掩盖真实的时序竞争 |
| SVA 断言编写 | 采样窗口错误，断言误报或漏报 |
| CDC 验证 | 无法正确判断同步器是否满足建立/保持时间 |
| FPGA 原型验证 | 忽略 I/O 时序约束，上板失败 |
| 签核（Signoff）配合 | 无法与 STA 工程师有效沟通违例问题 |

### 时序分析的目标

```
确保数据在正确的时间到达正确的位置
```

核心目标：
1. **建立时间满足** -- 数据在时钟沿到来前稳定
2. **保持时间满足** -- 数据在时钟沿到来后继续保持稳定
3. **时钟质量达标** -- 抖动（Jitter）和偏斜（Skew）在容忍范围内
4. **I/O 时序满足** -- 与外部器件的接口时序兼容

## 2. 时序路径分析

### 基本时序路径图

```
  ┌─────────┐    Tcq     ┌─────────┐   Tlogic    ┌─────────┐  Trouting  ┌─────────┐
  │  FF_src  │──────────>│ 逻辑锥   │───────────>│  组合逻辑 │──────────>│ FF_dest  │
  │  (源寄存器)│           │(Logic Cone)│            │(Comb Logic)│           │(目的寄存器)│
  └────┬─────┘           └─────────┘            └─────────┘           └────┬─────┘
       │                                                                    │
       │                           Tclk (时钟周期)                          │
       └────────────────────────────────────────────────────────────────────┘
                              时钟网络
```

### 时序公式

建立时间约束（最常见）：

```
Tclk > Tcq + Tlogic + Trouting + Tsu
```

| 符号 | 含义 | 典型值（28nm工艺） |
|------|------|-------------------|
| `Tclk` | 时钟周期 | 10ns（100MHz）~ 1ns（1GHz） |
| `Tcq` | 寄存器 Clock-to-Q 延迟 | 0.1 ~ 0.5ns |
| `Tlogic` | 组合逻辑传播延迟 | 取决于逻辑深度 |
| `Trouting` | 布线延迟 | 0.1 ~ 2ns（取决于布局） |
| `Tsu` | 目的寄存器建立时间 | 0.02 ~ 0.1ns |

时序裕量（Slack）计算：

```
Setup Slack = Tclk - (Tcq + Tlogic + Trouting + Tsu)
Hold  Slack  = (Tcq + Tlogic + Trouting) - Thold
```

> [!important] Slack 为正表示时序满足，为负表示时序违例

### 时序违例场景

```
场景1：组合逻辑过深
───────────────────────────────────────────────
Tclk   = 5ns
Tcq    = 0.3ns
Tlogic = 4.0ns  ← 逻辑级数过多
Trouting = 0.5ns
Tsu    = 0.1ns
───────────────────────────────────────────────
需求: 5ns > 0.3 + 4.0 + 0.5 + 0.1 = 4.9ns
Slack = 5 - 4.9 = 0.1ns（勉强满足，裕量不足）

场景2：布线延迟过大
───────────────────────────────────────────────
Tclk   = 2ns
Tcq    = 0.2ns
Tlogic = 0.8ns
Trouting = 1.2ns  ← 布线距离过长
Tsu    = 0.05ns
───────────────────────────────────────────────
需求: 2ns > 0.2 + 0.8 + 1.2 + 0.05 = 2.25ns
Slack = 2 - 2.25 = -0.25ns（违例！）
```

## 3. 建立时间分析

### 建立时间要求

建立时间（Setup Time）：数据必须在时钟有效沿到来**之前**保持稳定的最小时间。

```
时钟 CLK:     ┌───┐   ┌───┐   ┌───┐
              │   │   │   │   │   │
          ────┘   └───┘   └───┘   └───
                    ↑
                  采样沿

数据 DATA:    ═════╗   ╔═══════════
                   ║   ║
                   ╚═══╝
                   ↑   ↑
              Tsu窗口  采样沿

          |← Tsu →|
          数据必须在此窗口前稳定
```

> [!note] 建立时间是寄存器的固有属性，由工艺决定，典型值 20~100ps

### 建立时间检查（SVA 代码）

```systemverilog
// 方法1：使用蕴含操作符检查建立时间
property setup_check;
    @(posedge clk)
    // 在时钟沿前 Tsu 时间内，数据不应变化
    $stable(data_in) |-> $stable(data_in);
endproperty

// 方法2：更实用的建立时间窗口检查
// 在时钟沿采样时，检查数据在前一个周期是否稳定
property setup_window_check;
    @(posedge clk)
    // 如果在时钟沿采样到有效数据
    (valid) |-> (
        // 数据在上一个时钟沿到当前时钟沿之间保持稳定
        $stable(data_in) throughout (1)
    );
endproperty

// 方法3：通过时钟偏移模拟建立时间检查
module setup_checker #(
    parameter TSU = 0.05  // 50ps 建立时间
)(
    input logic clk,
    input logic data_in
);
    // 在 Tsu 之前的时刻采样数据
    logic data_sampled;
    always @(posedge clk) begin
        #(TSU);
        data_sampled <= data_in;
    end

    // 在时钟沿检查数据是否与 Tsu 前采样的一致
    assert property (
        @(posedge clk) data_in === data_sampled
    ) else $error("Setup violation: data changed within Tsu window");
endmodule
```

### 建立时间违例原因

| 原因 | 描述 | 修复方法 |
|------|------|---------|
| 组合逻辑过深 | 逻辑级数多，延迟大 | 插入流水线寄存器、优化逻辑 |
| 时钟频率过高 | Tclk 太小 | 降频、多周期路径约束 |
| 布线延迟大 | 物理距离远 | 优化布局、添加缓冲器 |
| 跨时钟域 | 异步时钟间数据传输 | 插入同步器、异步 FIFO |
| 高扇出网络 | 一个信号驱动大量负载 | 复制寄存器、插入缓冲树 |

## 4. 保持时间分析

### 保持时间要求

保持时间（Hold Time）：数据必须在时钟有效沿到来**之后**继续保持稳定的最小时间。

```
时钟 CLK:     ┌───┐   ┌───┐   ┌───┐
              │   │   │   │   │   │
          ────┘   └───┘   └───┘   └───
                    ↑
                  采样沿

数据 DATA:    ═══════════╗   ╔═══════
                         ║   ║
                         ╚═══╝
                    ↑    ↑
                  采样沿  Thold窗口

                       |←Th→|
              数据必须在此窗口后保持稳定
```

> [!warning] 保持时间违例比建立时间违例更危险 -- 它无法通过降频解决

### 保持时间检查（SVA 代码）

```systemverilog
// 方法1：检查数据在时钟沿后保持稳定
property hold_check;
    @(posedge clk)
    // 在时钟沿后 Thold 时间内，数据不应变化
    1 |-> ##0 $stable(data_in);
endproperty

// 方法2：使用采样窗口检查
// 在时钟沿采样后，数据不应立即改变
property hold_window_check;
    @(posedge clk)
    (valid) |-> (
        // 采样后的一个小时间窗口内数据稳定
        ##1 $stable(data_in)
    );
endproperty

// 方法3：通过延迟采样模拟保持时间检查
module hold_checker #(
    parameter THOLD = 0.02  // 20ps 保持时间
)(
    input logic clk,
    input logic data_in
);
    logic data_at_edge;
    logic data_after_hold;

    // 在时钟沿采样
    always @(posedge clk) begin
        data_at_edge <= data_in;
    end

    // 在 Thold 之后采样
    always @(posedge clk) begin
        #(THOLD);
        data_after_hold <= data_in;
    end

    // 两者应该一致，否则保持时间违例
    assert property (
        @(posedge clk) data_at_edge === data_after_hold
    ) else $error("Hold violation: data changed within Thold window");
endmodule
```

### 保持时间违例原因

| 原因 | 描述 | 修复方法 |
|------|------|---------|
| 最短路径过短 | 数据到达太快 | 插入延迟单元（Delay Cell） |
| 时钟偏斜过大 | 发射时钟快、捕获时钟慢 | 平衡时钟树 |
| 工艺角偏差 | 快工艺角下延迟减小 | 后端插入缓冲器 |
| 数据路径变化大 | 最长/最短路径差异大 | 路径平衡设计 |

> [!tip] 保持时间违例的黄金修复方法：在数据路径上插入延迟缓冲器（Delay Buffer），增加最小延迟

## 5. 时钟偏斜分析

### 时钟偏斜的产生

时钟偏斜（Clock Skew）：同一时钟信号到达不同寄存器时钟端的时间差。

```
                    ┌─────────────┐
                    │  时钟源 PLL  │
                    └──────┬──────┘
                           │
                    ┌──────┴──────┐
                    │  时钟树网络   │
                    └──┬───────┬──┘
                       │       │
              ┌────────┴──┐ ┌──┴────────┐
              │ Tclk1=1.2ns│ │ Tclk2=1.5ns│
              └─────┬─────┘ └─────┬─────┘
                    │             │
               ┌────┴────┐  ┌────┴────┐
               │  FF_src  │  │ FF_dest  │
               └─────────┘  └─────────┘

   Clock Skew = Tclk2 - Tclk1 = 1.5 - 1.2 = 0.3ns
```

### 时钟偏斜的影响

**对建立时间的影响（正偏斜有利）：**

```
Setup: Tclk > Tcq + Tlogic + Trouting + Tsu - Tskew

正偏斜（目的地时钟晚到）→ 等效增加时间裕量
负偏斜（目的地时钟早到）→ 减少时间裕量
```

**对保持时间的影响（正偏斜有害）：**

```
Hold: Tcq + Tlogic + Trouting > Thold + Tskew

正偏斜 → 数据可能被下一级过早捕获，增加保持违例风险
```

| 偏斜类型 | 建立时间 | 保持时间 |
|---------|---------|---------|
| 正偏斜（目的晚到） | 改善 | 恶化 |
| 负偏斜（目的早到） | 恶化 | 改善 |
| 零偏斜 | 基准 | 基准 |

### 时钟偏斜检查（SVA 代码）

```systemverilog
// 检查两个时钟域之间的偏斜
module skew_monitor #(
    parameter MAX_SKEW = 0.5  // 最大允许偏斜 500ps
)(
    input logic clk_a,
    input logic clk_b
);
    time clk_a_edge, clk_b_edge;
    real skew;

    always @(posedge clk_a) begin
        clk_a_edge = $time;
    end

    always @(posedge clk_b) begin
        clk_b_edge = $time;
        skew = clk_b_edge - clk_a_edge;
        if (skew < 0) skew = -skew;
        if (skew > MAX_SKEW) begin
            $warning("Clock skew = %0.2f ns exceeds limit %0.2f ns",
                     skew/1.0, MAX_SKEW/1.0);
        end
    end
endmodule

// 使用 SVA 检查时钟偏斜导致的数据竞争
property skew_data_check;
    @(posedge clk_dest)
    // 在目的时钟沿，源数据应已稳定
    $stable(src_data) |-> $stable(src_data);
endproperty
```

### 动态时序验证

通过仿真检查时序行为，需要测试向量驱动。

```systemverilog
// 动态时序验证示例：在仿真中检查建立/保持时间
module timing_check_example (
    input  logic clk,
    input  logic data_in,
    output logic data_out
);
    // 使用 $setuphold 系统任务进行时序检查
    // $setuphold(参考事件, 数据事件, Tsu, Thold, violation_flag)
    specify
        // 建立时间 50ps，保持时间 20ps
        $setuphold(posedge clk, data_in, 0.05, 0.02, notifier);
    endspecify

    always_ff @(posedge clk) begin
        data_out <= data_in;
    end
endmodule

// SDF 反标后的时序仿真
module sdf_timing_sim;
    initial begin
        // 加载标准延时格式文件
        $sdf_annotate("top_module.sdf", u_top,
                      ,              // log file
                      "MIN_TYP_MAX", // timing check
                      "1.0",         // scale factor
                      "1.0"          // MIPD scale
        );
    end
endmodule
```

### 形式时序验证

结合形式验证与时序分析，穷举证明时序属性。

```systemverilog
// 形式验证中的时序属性
module formal_timing_check (
    input  logic clk,
    input  logic rst_n,
    input  logic req,
    output logic gnt
);
    // 证明：请求发出后 3 个周期内必须得到应答
    property req_to_gnt_latency;
        @(posedge clk) disable iff (!rst_n)
        req |-> ##[1:3] gnt;
    endproperty

    assert property (req_to_gnt_latency)
        else $error("Grant not received within 3 cycles");

    // 证明：应答信号必须在请求后保持直到被确认
    property gnt_stable;
        @(posedge clk) disable iff (!rst_n)
        (req && !gnt) ##1 gnt |-> gnt until_with ack;
    endproperty

    assert property (gnt_stable);
endmodule
```

## 7. 时序违例调试

### 违例报告分析

**典型 Setup Violation 报告：**

```
*** VIOLATION ***
Startpoint: u_pipe/stage2/reg_data[7]
            (rising edge-triggered flip-flop clocked by clk_core)
Endpoint:   u_pipe/stage3/reg_data[7]
            (rising edge-triggered flip-flop clocked by clk_core)
Path Group: clk_core
Path Type:  max

  Delay    Time   Description
------------------------------------------------------------
  0.00    0.00   clock clk_core (rise edge)
  0.25    0.25   clock network delay (propagated)
  0.00    0.25   u_pipe/stage2/reg_data[7]/CK (DFF_X1)
  0.30    0.55   u_pipe/stage2/reg_data[7]/Q (DFF_X1)
  1.80    2.35   u_pipe/decode_logic/... (multistage logic)
  0.65    3.00   u_pipe/stage3/reg_data[7]/D (DFF_X1)
          3.00   data arrival time

  5.00    5.00   clock clk_core (rise edge)
  0.25    5.25   clock network delay (propagated)
 -0.10    5.15   clock uncertainty
 -0.08    5.07   library setup time
          5.07   data required time
------------------------------------------------------------
          5.07   data required time
         -3.00   data arrival time
------------------------------------------------------------
         -0.23   slack (VIOLATED)  ← 负裕量！违例 0.23ns
```

**报告解读：**

```
违例路径：stage2 → decode_logic → stage3
违例量：  -0.23ns（需要再快 0.23ns）
瓶颈：    decode_logic 组合逻辑延迟 1.80ns
```

### 违例修复方法

**Setup 违例修复策略（优先级从高到低）：**

```
1. 逻辑优化
   ├── 重组布尔表达式，减少逻辑级数
   ├── 使用查找表（LUT）替代多级逻辑
   └── 并行化关键路径逻辑

2. 流水线插入
   ├── 在组合逻辑中间插入寄存器
   ├── 增加 1 个时钟周期延迟
   └── 需要同步修改控制逻辑

3. 后端优化
   ├── 关键路径优先布局
   ├── 插入驱动缓冲器
   └── 调整单元尺寸（Cell Sizing）

4. 架构调整
   ├── 降低时钟频率
   ├── 多周期路径约束（MCP）
   └── 重新划分时钟域
```

**Hold 违例修复策略：**

```
1. 插入延迟单元
   ├── 在数据路径插入 Buffer
   ├── 使用专用 Delay Cell
   └── 后端工具自动修复

2. 时钟树调整
   ├── 调整时钟偏斜
   └── 重新平衡时钟树
```

**SDC 约束调整示例：**

```tcl
# 设置多周期路径（如果确实是多周期操作）
set_multicycle_path 2 -setup -from [get_cells u_pipe/stage2/*] \
                             -to   [get_cells u_pipe/stage3/*]
set_multicycle_path 1 -hold  -from [get_cells u_pipe/stage2/*] \
                             -to   [get_cells u_pipe/stage3/*]

# 设置虚假路径（如果路径不可达）
set_false_path -from [get_clocks clk_test] -to [get_clocks clk_core]

# 设置最大延迟约束
set_max_delay 3.5 -from [get_pins u_cpu/alu/*] -to [get_pins u_cpu/regfile/*]
```

## 8. 最佳实践

### 时序约束编写（SDC）

SDC（Synopsys Design Constraints）是业界标准的时序约束格式。

```tcl
# ============================================
# 完整的 SDC 约束示例
# ============================================

# 1. 时钟定义
create_clock -name clk_main -period 5.0 [get_ports clk]
create_clock -name clk_uart -period 20.0 [get_ports uart_clk]

# 2. 时钟不确定性（抖动 + 偏斜）
set_clock_uncertainty -setup 0.15 [get_clocks clk_main]
set_clock_uncertainty -hold  0.05 [get_clocks clk_main]

# 3. 时钟延迟
set_clock_latency -source 0.5 [get_clocks clk_main]
set_clock_latency 0.3 [get_clocks clk_main]

# 4. 输入延迟
set_input_delay -clock clk_main -max 1.5 [get_ports data_in*]
set_input_delay -clock clk_main -min 0.2 [get_ports data_in*]

# 5. 输出延迟
set_output_delay -clock clk_main -max 1.0 [get_ports data_out*]
set_output_delay -clock clk_main -min 0.1 [get_ports data_out*]

# 6. 跨时钟域约束
set_clock_groups -asynchronous -group [get_clocks clk_main] \
                               -group [get_clocks clk_uart]

# 7. 负载和驱动
set_load 0.05 [get_ports data_out*]
set_driving_cell -lib_cell BUFX4 [get_ports data_in*]

# 8. 多周期路径
set_multicycle_path 4 -setup -from [get_cells u_spi/*] -to [get_cells u_spi/*]
set_multicycle_path 3 -hold  -from [get_cells u_spi/*] -to [get_cells u_spi/*]

# 9. 虚假路径
set_false_path -from [get_clocks clk_scan]
```

### 时序验证策略

**分阶段验证流程：**

```
阶段1：RTL 阶段
├── SVA 断言检查基本时序属性
├── 时钟约束合理性审查
└── CDC 检查（SpyGlass/Conformal CDC）

阶段2：综合后
├── 初步 STA 报告分析
├── 关键路径识别
└── SDC 约束完善

阶段3：布局布线后
├── 详细 STA（含寄生参数）
├── 时钟树综合（CTS）后分析
└── 信号完整性分析（SI）

阶段4：签核
├── 多工艺角 STA（FF/TT/SS）
├── OCV（On-Chip Variation）分析
├── 时序 ECO 修复
└── 最终 Signoff
```

### 常见问题及解决

| 问题 | 现象 | 根因 | 解决方案 |
|------|------|------|---------|
| 仿真与 STA 结果不一致 | 仿真通过但 STA 违例 | SDF 未正确反标 | 检查 SDF 文件和反标流程 |
| 时钟域交叉违例 | CDC 路径大面积违例 | 跨时钟域未正确约束 | 使用 `set_clock_groups` |
| 多周期路径误报 | STA 报告大量虚假违例 | 路径缺少 MCP 约束 | 添加 `set_multicycle_path` |
| 保持违例集中出现 | 特定模块保持违例多 | 该模块时钟偏斜大 | 检查时钟树平衡 |
| 工艺角差异大 | FF/SS 下违例路径不同 | PVT 变化影响 | 多角分析、插入缓冲器 |
| I/O 时序违例 | 输入/输出路径违例 | 接口时序建模不准确 | 校准 `set_input/output_delay` |

> [!summary] 核心要点回顾
> 1. **建立时间**：数据必须在时钟沿前稳定，违例可通过降频或逻辑优化解决
> 2. **保持时间**：数据必须在时钟沿后保持稳定，违例只能通过增加延迟解决
> 3. **时钟偏斜**：对建立和保持时间的影响相反，需要在两者间权衡
> 4. **STA** 是签核级别的穷举分析，是时序验证的金标准
> 5. **SVA** 可用于动态时序检查，与 STA 互补
> 6. **SDC 约束**的质量直接决定时序分析的准确性
